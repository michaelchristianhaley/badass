BADASS — Report
What it actually is: not a coding project, but a governance constitution for an AI agent, encoded as a git repo. BADASS.md is a 354-line rulebook a user hands to an AI coding assistant (phone-driven Termux/PowerShell workflows, it seems) after that assistant has hallucinated, thrashed, or lied. The rest of the repo (control/, docs/, scripts/validate_repository.py) is scaffolding meant to make "the assistant read and complied with the rules" observable rather than a matter of trust.
What's great
The failure diagnosis is precise, not generic. Most "AI safety prompt" repos are vague ("be helpful and honest"). This one names concrete, real agentic-coding failure modes with real names: Thrash (abandoning a plan at the first sign of pushback), Drift (rabbit-holing into unrelated minutiae and losing the original goal), and HARD FAIL (stating unverified things as fact — including claiming to have run/tested/committed something that didn't happen). Anyone who has used an agentic coding assistant for more than a week will recognize all three immediately. That's a strong signal this was written from painful lived experience, not theory.
It attacks confabulation directly. The "Current State and Source Examination" section — no cached memory, no assumed-successful state, re-inspect live sources every operation — targets the single most damaging LLM behavior in long agentic sessions: quietly acting on stale or invented state.
The Thrash/question distinction is genuinely sharp. "Questions do not equal rejection" is a subtle, correct insight most alignment prompts miss — assistants very commonly reinterpret a user's clarifying question as "my plan was wrong, let me pivot," which is its own form of unreliability.
The outline-as-hard-cache pattern is a good design. Treating control/outline.md as a single source-of-truth "reality sandbox" that must be read before every operation and reconciled after is essentially a manual implementation of persistent agent state/memory — a reasonable workaround before tools had native memory.
Authority separation is enforceable in principle. User owns BADASS.md, the outline, and the inspection map; the assistant can "inspect and suggest" but never self-grant permissions. Decision records use explicit Proposed/Accepted/Superseded/Rejected statuses with archived (not deleted) history — that's straight out of the ADR (architecture decision record) playbook, applied to a very unusual domain.
What's weak or actually broken
control/inspection-map.yml is not YAML — it's JSON with a .yml extension. The file's entire raw content is { "version": 1, ... }. This is a small but real bug, and it's ironic: the project's core value proposition is "don't state false things as fact," and its own control file's name misrepresents its own format.
The whole system has no mechanical enforcement. This is the biggest gap. BADASS.md is prose instructing an LLM to behave a certain way — there's no hook, no system-prompt injection mechanism, no pre-commit or session-start automation described anywhere that guarantees the assistant even sees these rules on a given turn, let alone follows them. scripts/validate_repository.py validates that the files exist — it can't validate that an assistant actually walked the inspection gates or told the truth in a given session. A framework built to recover from an unreliable actor's failures relies entirely on that same actor voluntarily re-reading and self-applying the rules. That's the load-bearing weak point.
354 lines of dense "shall" statements is a lot to re-derive from attention every session. Nothing in the repo addresses how this text actually enters context each time (full paste into system prompt? Fetched by instruction only, which itself depends on the assistant remembering to fetch it?). Without a condensed, repeatable version, compliance will decay exactly the way long system prompts always do.
No worked example. For a repo whose entire purpose is "prove compliance is real," there's no sample bad-assistant transcript run through docs/RECOVERY-PROTOCOL.md and docs/SECTION-COMPLIANCE-MATRIX.md to demonstrate it actually produces a different outcome. Right now the framework is asserted, not demonstrated.
No license, on a public repo whose apparent secondary purpose is to be a reusable template. control/decisions/0004-license-selection.md leaves it unresolved, which blocks anyone else from legitimately forking this pattern for their own use — undercutting what looks like part of the point of publishing it.
Scope rigidity may backfire. "The User does not engage The Assistant as a friend... shall treat every prompt as formal work" is an absolute rule with no carve-out, applied by a system that elsewhere insists on precise, non-overreaching interpretation of intent. A rule this blunt will misfire on the (likely common) mixed conversation.
Improvements to match the latent intent
The underlying goal seems to be: a durable, evidence-based contract that makes any AI assistant this person works with (mostly via phone) stop lying about what it did, stop abandoning plans under mild pushback, and stay tethered to real state. To actually get there rather than just assert it on paper:

Fix inspection-map.yml → inspection-map.json, or rewrite its content as real YAML. Small, but it's the project eating its own dogfood.
Write an INTEGRATION.md explaining exactly how BADASS.md gets into an assistant's context each session (e.g., save as the tool's native project-instructions file — CLAUDE.md for Claude Code, .cursorrules, a pinned system prompt, etc.). This closes the single biggest gap in the whole design.
Ship a one-page distilled cheat-sheet (top ~10 rules) alongside the full BADASS.md, meant for repeated injection every turn, with the full doc remaining the canonical/legal reference. Long documents lose grip over long sessions; a short one is far more likely to actually get followed.
Wire up actual CI: a GitHub Action that runs scripts/validate_repository.py on push/PR, plus a status badge in the README, so "automated validation" is verifiably true rather than just described.
Add one worked example — a realistic bad-assistant transcript plus the recovery protocol and compliance matrix applied to it — so the framework's actual effect is demonstrable, not just aspirational.
Resolve the license decision so the repo can function as the reusable template it appears designed to be.
Version BADASS.md itself (front-matter with a version number / last-revised date), consistent with the decision-record discipline used everywhere else in the repo.
Narrow the "no casual conversation" line to technical/project work specifically, so the rest of the (well-designed) rigor doesn't get misapplied to unrelated conversation.

The core idea here is unusually good — most people trying to fix "my AI assistant keeps lying to me" write a paragraph of vibes. This person built a structured epistemic contract with real failure taxonomy and an audit trail. The gap is that it's a constitution with no enforcement mechanism and no demonstrated proof it works — closing that gap (integration doc, CI, and one worked example) would take it from "well-written aspiration" to "framework that provably does what it claims."
